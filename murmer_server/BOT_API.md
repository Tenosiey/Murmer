# Murmer Bot API

Murmer supports programmatic access through **bots** – lightweight accounts
that authenticate with a token instead of an Ed25519 keypair. Bots can read
and send messages, react to messages, list channels and users, and optionally
manage channels.

Because Murmer has no central server, bots are created and managed per-server.
Each Murmer instance maintains its own bot registry. A bot token is only valid
on the server that issued it.

---

## Quick start

### 1. Create a bot

Ask the server admin to create a bot. This requires the `ADMIN_TOKEN`
environment variable to be set on the server.

```bash
curl -X POST http://localhost:3001/api/v1/bots \
  -H "Authorization: Bearer <ADMIN_TOKEN>" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "GreeterBot",
    "permissions": ["read_messages", "send_messages", "read_channels"],
    "description": "Greets new users"
  }'
```

The response includes a `token` field – **save it**, it is only shown once:

```json
{
  "data": {
    "id": "a1b2c3d4e5f6...",
    "name": "GreeterBot",
    "permissions": ["read_messages", "send_messages", "read_channels"],
    "description": "Greets new users",
    "active": true,
    "created_at": "2026-03-10T12:00:00Z",
    "token": "mrm_abc123..."
  }
}
```

### 2. Send a message

```bash
curl -X POST http://localhost:3001/api/v1/channels/general/messages \
  -H "Authorization: Bearer mrm_abc123..." \
  -H "Content-Type: application/json" \
  -d '{"text": "Hello from GreeterBot!"}'
```

### 3. Read messages

```bash
curl http://localhost:3001/api/v1/channels/general/messages \
  -H "Authorization: Bearer mrm_abc123..."
```

## Concepts

### Tokens

Bot tokens use the format `mrm_<base64url>`. They are generated by the server
and stored as a SHA-256 hash in the database – the plaintext is never
persisted. If a token is lost, an admin can regenerate it with the
`/reset-token` endpoint.

### Permissions

Bots have fine-grained permissions represented as a bitfield internally and a
string list in the API:

| Permission | Description |
|---|---|
| `read_messages` | Read message history in any channel |
| `send_messages` | Send messages (and delete own messages) |
| `manage_messages` | Delete any user's messages |
| `add_reactions` | Add and remove emoji reactions |
| `read_channels` | List available channels |
| `manage_channels` | Create and delete channels (text & voice) |
| `read_users` | List online users and their statuses |

When creating a bot without specifying permissions, it defaults to
`read_messages` + `send_messages`.

### Bot messages

Messages sent by bots include a `"bot": true` field in the JSON payload.
Clients can use this to display a bot badge. The `user` field is set to the
bot's name.

### Decentralised model

Each Murmer server is independent. There is no central bot registry. If you
want your bot on multiple servers:

1. Ask each server's admin to create a bot account
2. Configure your bot with each server's URL and token
3. Your bot connects to each server independently

This mirrors Murmer's philosophy – no single point of control.

---

## Authentication

All API requests require a `Bearer` token in the `Authorization` header:

```
Authorization: Bearer mrm_abc123...
```

Admin management endpoints use the server's `ADMIN_TOKEN`. Bot API endpoints
use the bot's own token.

### Rate limiting

Bot message sending is rate-limited using the same sliding window as regular
users (default: 30 messages/minute, configurable via
`MAX_MESSAGES_PER_MINUTE`). Other read endpoints are not rate-limited.

## Admin endpoints

These endpoints manage bot accounts and require the `ADMIN_TOKEN` for
authentication.

### Create bot

```
POST /api/v1/bots
Authorization: Bearer <ADMIN_TOKEN>
```

**Body:**

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | string | yes | Display name (1-32 chars, alphanumeric/dash/underscore/space) |
| `owner_key` | string | no | Public key of the bot's owner (for accountability) |
| `permissions` | string[] | no | Permission list (defaults to `["read_messages", "send_messages"]`) |
| `description` | string | no | Short description (max 256 chars) |

**Response:** `201 Created`

```json
{
  "data": {
    "id": "a1b2c3d4...",
    "name": "MyBot",
    "owner_key": "",
    "permissions": ["read_messages", "send_messages"],
    "description": "",
    "active": true,
    "created_at": "2026-03-10T12:00:00Z",
    "token": "mrm_<secret>"
  }
}
```

The `token` field is only included in the creation response.

### List bots

```
GET /api/v1/bots
Authorization: Bearer <ADMIN_TOKEN>
```

**Response:** `200 OK`

```json
{
  "data": [
    {
      "id": "a1b2c3d4...",
      "name": "MyBot",
      "permissions": ["read_messages", "send_messages"],
      "description": "",
      "active": true,
      "created_at": "2026-03-10T12:00:00Z"
    }
  ]
}
```

### Get bot

```
GET /api/v1/bots/:bot_id
Authorization: Bearer <ADMIN_TOKEN>
```

### Update bot

```
PATCH /api/v1/bots/:bot_id
Authorization: Bearer <ADMIN_TOKEN>
```

**Body:** (all fields optional)

| Field | Type | Description |
|---|---|---|
| `name` | string | New display name |
| `permissions` | string[] | New permission set (replaces existing) |
| `description` | string | New description |
| `active` | boolean | Enable/disable the bot |

Setting `active` to `false` immediately invalidates the bot's token for all
future requests without deleting the bot.

### Delete bot

```
DELETE /api/v1/bots/:bot_id
Authorization: Bearer <ADMIN_TOKEN>
```

**Response:** `204 No Content`

### Reset token

```
POST /api/v1/bots/:bot_id/reset-token
Authorization: Bearer <ADMIN_TOKEN>
```

Generates a new token and invalidates the old one. Returns the new token:

```json
{
  "data": {
    "bot_id": "a1b2c3d4...",
    "token": "mrm_<new_secret>"
  }
}
```

## Bot API endpoints

These endpoints are used by the bot itself to interact with the server.

### Server info

```
GET /api/v1/server/info
Authorization: Bearer <bot_token>
```

Returns server metadata. Requires any valid bot token (no specific permission).

```json
{
  "data": {
    "version": "2026.3.6-alpha.1",
    "bot_api_version": "1",
    "online_users": 5,
    "channels": ["general", "random"],
    "has_password": false
  }
}
```

### List channels

```
GET /api/v1/channels
Authorization: Bearer <bot_token>
```

**Permission required:** `read_channels`

```json
{
  "data": {
    "channels": ["general", "random", "dev"]
  }
}
```

### Read messages

```
GET /api/v1/channels/:channel/messages
Authorization: Bearer <bot_token>
```

**Permission required:** `read_messages`

**Query parameters:**

| Param | Type | Default | Description |
|---|---|---|---|
| `limit` | integer | 50 | Number of messages (1-200) |
| `before` | integer | – | Fetch messages older than this ID (descending cursor) |
| `after` | integer | – | Fetch messages newer than this ID (ascending cursor) |

Messages are always returned in ascending (chronological) order.

```json
{
  "data": {
    "channel": "general",
    "messages": [
      {
        "id": 42,
        "type": "chat",
        "user": "Alice",
        "text": "Hello!",
        "timestamp": "2026-03-10T12:00:00+00:00",
        "time": "12:00:00",
        "channel": "general",
        "reactions": {"👍": ["Bob"]},
        "bot": false
      }
    ],
    "has_more": true
  }
}
```

**Polling pattern:** To continuously watch for new messages, save the highest
`id` you've seen and poll with `?after=<id>`:

```bash
# Initial fetch
curl ".../channels/general/messages?limit=10"

# Subsequent polls (replace 42 with highest seen id)
curl ".../channels/general/messages?after=42"
```

### Send message

```
POST /api/v1/channels/:channel/messages
Authorization: Bearer <bot_token>
```

**Permission required:** `send_messages`

**Body:**

| Field | Type | Required | Description |
|---|---|---|---|
| `text` | string | yes | Message content (1-4000 chars) |
| `ephemeral` | boolean | no | Auto-delete after expiry (default: false) |
| `expires_in_seconds` | integer | no | Seconds until auto-delete (5-86400, default: 60) |

**Response:** `201 Created`

```json
{
  "data": {
    "id": 100,
    "type": "chat",
    "user": "GreeterBot",
    "text": "Hello!",
    "timestamp": "2026-03-10T12:05:00+00:00",
    "channel": "general",
    "bot": true,
    "reactions": {}
  }
}
```

The message is immediately broadcast to all connected WebSocket clients in the
channel.

### Delete message

```
DELETE /api/v1/channels/:channel/messages/:message_id
Authorization: Bearer <bot_token>
```

**Permission required:** `send_messages` for own messages, `manage_messages`
for other users' messages.

**Response:** `204 No Content`

### Add reaction

```
POST /api/v1/channels/:channel/messages/:message_id/reactions
Authorization: Bearer <bot_token>
```

**Permission required:** `add_reactions`

**Body:**

```json
{"emoji": "👍"}
```

### Remove reaction

```
DELETE /api/v1/channels/:channel/messages/:message_id/reactions/:emoji
Authorization: Bearer <bot_token>
```

**Permission required:** `add_reactions`

The emoji in the URL should be percent-encoded if it contains special
characters.

### List users

```
GET /api/v1/users
Authorization: Bearer <bot_token>
```

**Permission required:** `read_users`

```json
{
  "data": {
    "online": ["Alice", "Bob"],
    "all": ["Alice", "Bob", "Charlie"],
    "statuses": {
      "Alice": "online",
      "Bob": "away",
      "Charlie": "offline"
    }
  }
}
```

## WebSocket connection

Bots can also connect via WebSocket for real-time event streaming. This is
useful for bots that need to react to messages as they arrive rather than
polling.

### Connecting

Connect to `ws://<server>/ws` and send a `bot-presence` message:

```json
{
  "type": "bot-presence",
  "token": "mrm_abc123..."
}
```

On success the server sends initial state (channel list, user list, history)
and begins streaming events. The bot receives all the same events as a regular
client: `chat`, `message-deleted`, `reaction-update`, `channel-add`,
`channel-remove`, `online-users`, `status-update`, etc.

### Sending messages via WebSocket

Once authenticated, bots can send messages the same way as regular clients:

```json
{
  "type": "chat",
  "user": "GreeterBot",
  "text": "Hello everyone!",
  "timestamp": "2026-03-10T12:00:00Z"
}
```

The server broadcasts the message to the channel and persists it. Note that
WebSocket messages from bots are not automatically tagged with `"bot": true` –
use the REST API if you need that tag.

### Keepalive

Send periodic ping messages to keep the connection alive:

```json
{"type": "ping", "id": "1"}
```

The server responds with:

```json
{"type": "pong", "id": "1"}
```

## Error responses

All errors follow a consistent format:

```json
{"error": "error-code"}
```

| HTTP Status | Error Code | Description |
|---|---|---|
| 400 | `invalid-bot-name` | Bot name is empty, too long, or contains invalid characters |
| 400 | `invalid-channel-name` | Channel name validation failed |
| 400 | `invalid-message-text` | Message text is empty or exceeds 4000 characters |
| 400 | `invalid-message-id` | Message ID is not a valid integer |
| 400 | `invalid-emoji` | Emoji is empty, too long, or contains invalid characters |
| 400 | `description-too-long` | Bot description exceeds 256 characters |
| 401 | `invalid-admin-token` | The provided admin token is incorrect |
| 401 | `invalid-bot-token` | The bot token is invalid or the bot is deactivated |
| 403 | `missing-permission:*` | The bot lacks the required permission |
| 404 | `bot-not-found` | No bot with the given ID exists |
| 404 | `channel-not-found` | No channel with the given name exists |
| 404 | `message-not-found` | No message with the given ID exists in that channel |
| 429 | `rate-limit-exceeded` | Too many messages sent in the time window |
| 500 | various | Internal server error |

## Building a bot

Bots are just HTTP clients. You can write them in any language. Here is a
minimal example in Python:

```python
import requests
import time

SERVER = "http://localhost:3001"
TOKEN = "mrm_abc123..."
HEADERS = {
    "Authorization": f"Bearer {TOKEN}",
    "Content-Type": "application/json",
}

def send_message(channel, text):
    r = requests.post(
        f"{SERVER}/api/v1/channels/{channel}/messages",
        headers=HEADERS,
        json={"text": text},
    )
    r.raise_for_status()
    return r.json()

def poll_messages(channel, after=None):
    params = {"limit": 50}
    if after:
        params["after"] = after
    r = requests.get(
        f"{SERVER}/api/v1/channels/{channel}/messages",
        headers=HEADERS,
        params=params,
    )
    r.raise_for_status()
    return r.json()["data"]

# Simple echo bot
last_id = None
while True:
    data = poll_messages("general", after=last_id)
    for msg in data["messages"]:
        last_id = msg["id"]
        user = msg.get("user", "")
        text = msg.get("text", "")
        if msg.get("bot"):
            continue  # ignore other bots
        if text.startswith("!echo "):
            send_message("general", text[6:])
    time.sleep(2)
```

### Tips

- **Poll responsibly.** A 2-5 second interval is usually sufficient. For
  lower latency, use the WebSocket connection instead.
- **Handle rate limits.** If you get a 429 response, back off and retry.
- **Check permissions.** Use `/api/v1/server/info` on startup to verify the
  bot can reach the server and see which channels are available.
- **Tag your messages.** The REST API automatically tags messages with
  `"bot": true` so clients can identify bot messages.
- **Respect the server.** Bots share rate limits with human users. Avoid
  spamming or flooding channels.

## Adding a bot to your server

1. Set the `ADMIN_TOKEN` environment variable when starting the server:

   ```bash
   ADMIN_TOKEN=my-secret-admin-token cargo run
   ```

2. Create a bot account using the admin API (see [Create bot](#create-bot)).

3. Give the bot token to the person or script that will run the bot.

4. The bot authenticates with its token and can immediately interact with the
   server.

5. To disable a bot, update it with `"active": false` or delete it entirely.

---

## API versioning

The current API version is **v1** (`/api/v1/...`). Future breaking changes
will be introduced under `/api/v2/` while keeping v1 available for a
transition period. The `bot_api_version` field in `/api/v1/server/info`
indicates the latest supported version.
